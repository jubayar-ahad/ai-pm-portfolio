# `tool-use-agent/fixtures/`

Tiny, build-root-level fixture data shipped alongside the package so the
`sql_query` tool (and any future structured-data tool) has something to
query without external setup.

## `sample.db`

A 20 KB SQLite database with three tables — `authors`, `books`,
`book_genres` — designed so the `sql_query` tool can demonstrate the
shape of operations agents commonly need:

- Single-table `SELECT … WHERE` (filtering)
- `JOIN` across tables (authors ↔ books)
- `GROUP BY … COUNT(*)` (aggregation)
- `LIMIT` (with the tool's `max_rows` cap as a redundant guardrail)

Schema and rows are reproducibly generated by `make_sample_db.py`. The
binary is committed so a fresh checkout can run the demo without first
running the generator. Regenerate with:

    python3 tool-use-agent/fixtures/make_sample_db.py

The file pins `PRAGMA application_id = 0x54554153` ("TUAS") so an
external `file(1)` / `sqlite3 .dbinfo` check can confirm the fixture
identity without parsing the contents.

## Why build-root, not `tests/fixtures/`

NEXT_WORK item 6 sub-checkbox 1 specifies `tool-use-agent/fixtures/`
literally — the fixture is for the tool's *users* (the agent, the CLI,
the demo invocation), not just for the pytest suite. The package's
`pyproject.toml` excludes `tests/` from the built wheel; the
build-root `fixtures/` directory is where data the package's runtime
demos depend on belongs.
